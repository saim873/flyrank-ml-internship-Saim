01 · Abstract

A ranked signal for editorial teams, not an automated editor

Content teams managing large catalogs of published pages face a recurring question: which pages are worth revisiting, and which are safe to leave alone? Manual review does not scale past a few hundred URLs, and simple heuristics — such as "revisit anything older than a year" — ignore the search-performance evidence that is already available in most analytics stacks.

This study formulates the question as a supervised classification problem. Using anonymized, historical search-performance signals — impressions, click-through rate, average position, sessions, and content-age indicators — a Random Forest classifier is trained to estimate whether a page is a refresh candidate. The model is evaluated under two validation regimes: a random split, which permits information from a single client's pages to appear in both training and testing, and a grouped split, which holds entire clients out of training. The grouped result (accuracy 0.796, F1 0.840) is reported as the honest estimate of performance, since it reflects how the model will actually be used — on clients it has not seen before. The output of the pipeline is a ranked recommendation queue with reason codes, intended to support — not replace — a human editorial decision.

02 · Introduction

Why content prioritization matters

The business problem

Search visibility decays gradually and unevenly across a content catalog. Some pages lose impressions as competitors publish fresher material; others simply age out of relevance. Editorial teams have limited time and cannot manually audit every page on a schedule, so the pages that most need attention often go unreviewed until visibility has already dropped substantially.

Why prioritization, not automation

Refreshing content is a judgment call that depends on context a model does not have — brand voice, legal review, seasonal relevance, and business priorities. The goal of this project is to make that judgment call more efficient, by surfacing a small, ranked set of candidates with clear reasoning, so a human reviewer spends time where it is most likely to matter.

03 · Problem Statement

Scope, unit of analysis, and the cost of being wrong

Decision supported
Whether a given page should be placed into an editorial review queue for a possible content refresh, within the current reporting window.
Unit of analysis
A single content URL, described by its own search-performance history over recent and trailing windows (30-day, 90-day) plus structural attributes such as word count and content age.
Output
A binary refresh-candidate label with an associated probability, converted into a ranked queue and a priority tier (Refresh, Monitor, No Action) with a reason code.
False positive
A page is flagged as a refresh candidate but did not actually need one. Cost: reviewer time spent on a low-value page. Recoverable — the reviewer simply moves on.
False negative
A page that should have been reviewed is not flagged. Cost: a genuine visibility decline continues unaddressed until the next scoring cycle or a manual audit catches it.
Human review
Every flagged page passes through a human decision point before any content, metadata, or publishing action is taken. The model narrows the queue; it does not act on it.

05 · Data Safety & Leakage Controls

What the model is not allowed to see

Several columns available in the raw dataset were deliberately excluded from the feature set because they could leak the answer, leak client identity, or leak information from the future relative to the prediction point.

Excluded columns

  • content_id — a unique identifier that could allow the model to memorize specific pages rather than learn generalizable patterns.
  • client_id — direct client identity; retaining it risks the model learning "which client" instead of "which signal pattern."
  • trend_direction — a derived summary of the outcome trend, close enough to the label itself to constitute leakage.
  • trend_pct — a numeric trend measure computed over a window that overlaps with the prediction target.

Why this matters

Identifier columns let a model memorize training examples instead of generalizing. Derived trend columns are computed from the same underlying signal as the label, so including them would produce inflated, misleading accuracy. Removing all four columns before training keeps the reported metrics honest and keeps the model transferable to clients it has never seen.

07 · Rule-based baseline

Before training a model, a transparent rule-based baseline was implemented to establish a reference point and to sanity-check that the problem is not trivially solvable with a simple heuristic. The baseline, referred to as the Week-4 rule, scores a page using its trailing impression trend and average position: pages showing a declining impression trend combined with a mid-to-low ranking position are flagged as refresh candidates. The rule is intentionally simple and interpretable, and it is evaluated on the identical held-out data used for the machine learning model, so that the comparison in the Results section is apples-to-apples.

10 · Machine learning model

The model is a Random Forest classifier configured with 200 trees and balanced class weighting, so that the more frequent class does not dominate learning. Missing numeric values are imputed with the column median, and categorical fields are one-hot encoded prior to fitting. A fixed random state of 42 is used throughout data splitting and model fitting to keep results reproducible across runs.

  • 200 decision trees, balanced class weight
  • Median imputation for missing numeric features
  • One-hot encoding for categorical features
  • Fixed random state (42) for reproducibility
08 · Validation Design

Grouped validation is the honest number

Two validation regimes were run side by side specifically to audit whether client-level leakage was inflating the model's apparent performance.

Random split

Rows are shuffled and split into train and test sets without regard to which client they belong to. Because a single client can contribute many pages, pages from the same client can end up in both the training set and the test set. The model can then partly succeed by recognizing client-specific patterns it already saw during training, rather than genuinely generalizing to new clients.

Grouped split

Clients — not rows — are assigned entirely to either the training set or the test set. No client appears in both. This forces the model to be evaluated on clients it has never encountered, which mirrors how the system will actually be used in production: scoring content from a client whose data was not part of training.

Random split

Leakage present

Client C has pages on both sides of the split, so the model may already recognize its patterns before being "tested" on it.

Grouped split

No leakage

Every client sits entirely inside one box. None of the 7 held-out test clients were seen during training on the other 24.

Why grouped validation is reported as the honest estimate

The random split produced a higher accuracy (0.841) than the grouped split (0.796). That gap is evidence of client leakage rather than genuinely better performance: some of the random split's apparent skill came from having already seen that client's pages. Because the model will be applied to new clients in practice, the grouped-split metrics are treated as the reliable, decision-relevant estimate of performance throughout this study.

13 · Error Analysis

Understanding the misclassifications

On the grouped test set of 5,098 pages, the model produced 1,041 incorrect predictions. Reviewing these cases suggests several contributing, non-exclusive factors.

Missing values

Pages with sparse historical data force reliance on imputed medians, reducing the precision of the signal for that record.

Low-traffic pages

Pages with few impressions produce noisy rate-based features such as click-through rate, making the underlying trend harder to estimate reliably.

Seasonality

Content tied to seasonal demand can show temporary dips or spikes in impressions that resemble a genuine refresh signal without actually being one.

Complex interactions

Some pages sit at the boundary between categories, where multiple signals interact in ways a single decision threshold cannot cleanly separate.

14 · Recommendations Engine

A ranked, reason-coded queue for editorial review

Model probabilities are converted into a ranked list with a priority tier and a plain-language reason code, so a reviewer can act quickly without inspecting raw feature values.

Every row that reaches Refresh or Monitor status is routed to a Human Review queue before any change is made to the underlying content.

15 · Human Review Rules

What this system is never allowed to do on its own

Never auto-publish

The model does not edit, rewrite, or publish content. It only produces a ranked list and a reason code for a human editor to act on.

Never auto-delete

Pages flagged as low-priority or stable are not removed or unpublished automatically. Removal decisions remain entirely with the editorial team.

Never assume causation

A flagged decline is an observed association between signals and outcomes, not evidence that any specific factor caused the decline.

16 · Monitoring & Retraining Triggers

Keeping the model honest over time

Data drift

Monitor the distribution of incoming feature values against the training distribution. A material shift in impressions, position, or content-age patterns is a signal to re-evaluate the model.

Performance drift

Track precision and recall on newly reviewed outcomes. A sustained drop below the grouped-validation baseline (F1 0.840) is a retraining trigger.

Traffic changes

Broad shifts in overall search traffic — seasonal or platform-driven — should prompt a review of whether the current feature set and thresholds still hold.

17 · Limitations

What this study does and does not claim

This study reports

  • Observed associations between search-performance signals and refresh outcomes in the historical dataset.
  • Measured accuracy, precision, recall, and F1 under a leakage-controlled, grouped validation split.
  • Directional guidance — a ranked, relative ordering of which pages are more likely to benefit from review.
  • Decision-support output intended to inform, not replace, editorial judgment.

This study does not claim

  • That any signal causes a change in search visibility.
  • Any knowledge of, or relationship to, Google's ranking algorithm or its internal mechanics.
  • That the model's recommendations are guaranteed outcomes for any individual page.
  • That performance observed on this dataset will transfer unchanged to every future client or content type.
